## 3. 成本估算（UCP用例点估算法）

### 3.1 未调整用例点（UUCP）

如**表7.2**所示，8个用例按复杂度分类，UUCP合计75点。

**表7.2 UUCP计算
| --- | --- | --- | --- | --- |
| 简单（≤3事务） | UC01浏览地图、UC02查询景点、UC07导出数据 | 3 | 5 | 15 |
| 中等（4-7事务） | UC03 AI问答、UC05管理景点、UC08录制导航 | 3 | 10 | 30 |
| 复杂（>7事务） | UC04规划路线、UC06导入数据 | 2 | 15 | 30 |
| **UUCP合计** | | **8** | | **75** |

### 3.2 技术复杂度因子（TCF）

如**表7.3**所示，13个技术因子加权求和后计算TCF=0.95。

**表7.3 TCF计算
| --- | --- | --- | --- | --- |
| T1 | 分布式系统 | 2.0 | 1 | 2 |
| T2 | 响应时间 | 1.0 | 3 | 3 |
| T3 | 最终用户效率 | 1.0 | 3 | 3 |
| T4 | 内部处理复杂度 | 1.0 | 3 | 3 |
| T5 | 代码复用性 | 1.0 | 4 | 4 |
| T6 | 易安装性 | 0.5 | 4 | 2 |
| T7 | 易使用性 | 0.5 | 4 | 2 |
| T8 | 可移植性 | 2.0 | 2 | 4 |
| T9 | 易变更性 | 1.0 | 3 | 3 |
| T10 | 并发性 | 1.0 | 2 | 2 |
| T11 | 安全性 | 1.0 | 3 | 3 |
| T12 | 第三方依赖 | 1.0 | 3 | 3 |
| T13 | 培训需求 | 1.0 | 1 | 1 |
| | **Σ（权重×取值）** | | | **35** |

**TCF = 0.6 + 0.01 × Σ = 0.6 + 0.35 = 0.95**

### 3.3 环境因子（EF）

如**表7.4**所示，8个环境因子加权求和后计算EF=1.025。

**表7.4 EF计算
| --- | --- | --- | --- | --- |
| E1 | UML模型熟悉度 | 1.5 | 3 | 4.5 |
| E2 | 应用领域经验 | 0.5 | 2 | 1.0 |
| E3 | 面向对象经验 | 1.0 | 3 | 3.0 |
| E4 | 组长能力 | 0.5 | 4 | 2.0 |
| E5 | 团队动力 | 1.0 | 3 | 3.0 |
| E6 | 需求稳定性 | 2.0 | 2 | 4.0 |
| E7 | 兼职人员 | -1.0 | 3 | -3.0 |
| E8 | 编程语言难度 | -1.0 | 2 | -2.0 |
| | **Σ（权重×取值）** | | | **12.5** |

**EF = 1.4 + (-0.03) × Σ = 1.4 - 0.375 = 1.025**

### 3.4 调整后用例点

**UCP = UUCP × TCF × EF = 75 × 0.95 × 1.025 = 73.0 用例点**

## 4. 风险预警

### 4.1 风险识别与评估

如**表7.5**所示，识别7项风险，其中1项高风险（R01 AI API不可用）、3项中风险、3项低风险。

**表7.5 风险识别与评估
| --- | --- | --- | --- | --- | --- | --- |
| R01 | AI API不可用（网络封锁/Key过期） | 技术 | 70% | 5 | 3.50 | 🔴 高 |
| R02 | 空间查询性能不足（大数据量） | 技术 | 40% | 3 | 1.20 | 🟡 中 |
| R03 | 坐标数据精度不足/数据错误 | 质量 | 50% | 3 | 1.50 | 🟡 中 |
| R04 | 项目进度延迟 | 进度 | 40% | 4 | 1.60 | 🟡 中 |
| R05 | 成员中途退出/参与度不足 | 资源 | 20% | 4 | 0.80 | 🟢 低 |
| R06 | 数据库文件损坏/数据丢失 | 技术 | 10% | 5 | 0.50 | 🟢 低 |
| R07 | jpackage/MSI构建失败 | 技术 | 30% | 3 | 0.90 | 🟢 低 |

### 4.2 概率影响矩阵

如**图7.9**所示，风险按概率×影响落在矩阵不同区域，R01居于高概率×高影响区域（不可接受），需优先应对。

```
影响\概率 │ 低(10-30%)  │ 中(40-50%)  │ 高(60-70%)
─────────┼──────────────┼──────────────┼─────────────
  高(4-5) │ R05, R06, R07│ R04         │ R01
  中(3)   │              │ R02, R03    │
  低(1-2) │              │             │
```

**风险等级**：🔴 高（不可接受）→ 🟡 中（需关注）→ 🟢 低（可接受）

**图7.9 概率影响矩阵**：高风险R01需优先制定缓解措施，低风险持续监控即可。

## 5. 空间数据维护方案

### 5.1 数据更新策略

如**表7.8**所示，数据更新按不同类型采用不同策略。

**表7.8 数据更新策略
| --- | --- | --- | --- |
| 新增亭子 | 管理员手动录入/批量导入 | 即时生效 | 无需 |
| 修改坐标 | 管理员编辑 | 即时生效 | 需二次确认 |
| 删除亭子 | 管理员删除 | 软删除（标记deleted_at） | 需输入原因 |
| 批量更新 | Excel/GeoJSON重新导入 | 覆盖+新增 | 管理员确认 |
| 坐标校正 | GCJ-02→WGS-84自动转换 | 批量作业 | 自动 |

### 5.2 数据质量控制

如**表7.9**所示，数据质量控制覆盖坐标范围、精度、必填字段和名称唯一性。

**表7.9 数据质量控制规则
| --- | --- | --- |
| 坐标范围 | lon: -180~180, lat: -90~90 | 超出拒绝写入 |
| 中国境内 | lat≥18°, lat≤54°, lon≥73°, lon≤135° | 超出标记警告 |
| 坐标精度 | WGS-84保留6位小数（~0.1m） | 自动round |
| 必填字段 | name, chinese_name, longitude, latitude | 空值拒绝写入 |
| 名称唯一性 | name字段UNIQUE约束 | 重复提示覆盖 |
| 数据来源 | collector_id 必须填写 | 空值拒绝 |
| 多媒体链接 | 检测URL可达性（HEAD请求） | 不可达标记warning |

### 5.3 备份策略

如**表7.10**所示，数据备份覆盖数据库、配置、日志三类。

**表7.10 备份策略
| --- | --- | --- | --- | --- |
| H2数据库(.mv.db) | 文件复制 | 每日00:00 | 7天 | backup/目录 |
| H2数据库 | 手动导出SQL Script | 每周 | 4周 | backup/sql/ |
| PostgreSQL | pg_dump | 每日 | 30天 | /var/backups/ |
| 配置文件(yml) | Git版本控制 | 每次提交 | 永久 | GitHub |
| 前端静态资源 | Git + CI制品 | 每次Release | 永久 | GitHub Releases |
| 日志文件 | 自动滚动 | 每天 | 30天 | logs/archive/ |

### 5.4 H2文件备份脚本

```batch
@echo off
chcp 65001 >nul
set DB_DIR=%userprofile%\tingcheng\db
set BACKUP_DIR=%userprofile%\tingcheng\backup
set DATE=%date:~0,4%%date:~5,2%%date:~8,2%

if not exist "%BACKUP_DIR%" mkdir "%BACKUP_DIR%"
copy "%DB_DIR%\tingcheng.mv.db" "%BACKUP_DIR%\tingcheng_%DATE%.mv.db"
echo 备份完成: tingcheng_%DATE%.mv.db

REM 保留最近7天，删除更早的
forfiles /p "%BACKUP_DIR%" /m *.mv.db /d -7 /c "cmd /c del @path" 2>nul
```

### 5.5 Docker Volume备份

```bash
# 停止容器
docker-compose down

# 备份Volume
docker run --rm -v pgdata:/source -v $(pwd)/backup:/dest alpine \
  tar czf /dest/postgres-backup-$(date +%Y%m%d).tar.gz -C /source .

# 恢复
docker run --rm -v pgdata:/dest -v $(pwd)/backup:/source alpine \
  tar xzf /source/postgres-backup-20260601.tar.gz -C /dest/
```

### 5.6 数据追溯机制（collector_id）

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| collector_user_id | 采集人学号/工号 | "20240701" |
| collector_name | 采集人姓名 | "张三" |
| collection_time | 采集时间 | "2026-05-15 14:30:00" |
| collection_tool | 采集工具 | "手机GPS / 手持GPS / 百度地图" |
| accuracy | 定位精度（米） | "5m" |
| data_source | 数据来源 | "实地采集 / 网络公开数据 / 文献整理" |

每位成员通过 `POST /thousand-pavilions/{id}/collectors` 填写采集记录，数据提交后写入 `pavilion_collector` 表，支持按 collector_user_id 筛选与导出。
